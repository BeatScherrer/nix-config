Agentic Development Loop

Concept & Implementation Plan
Part 1

Concept

The Problem

Traditional remote development requires a full environment: terminal, editor, keyboard. Working from a phone means awkward typing, limited screen space, and no access to the toolchain. Even with SSH and tmux, you're manually driving every action.

The Solution

Shift the developer's role from operator to director. Instead of writing code character by character, the developer issues intent — what should happen and why — and a system of specialized agents executes the work, reports progress, and asks for approval when needed.

The Loop

Intent → Plan → Code → Review → Test → Deploy → Monitor Developer (Element / Matrix on any device) │ ▼ Orchestrator (ZeroClaw instance on Matrix) │ ← receives messages natively via ZeroClaw's Matrix channel │ ← custom skills: spawn agents, route tasks, manage state │ ├── Planner ZeroClaw instance + planner skills ├── Coder ZeroClaw instance + coder skills + file/git tools ├── Reviewer ZeroClaw instance + reviewer skills + diff tools ├── Tester ZeroClaw instance + test runner tools ├── Deployer ZeroClaw instance + CI/deploy tools └── Monitor ZeroClaw daemon + log/metrics tools │ ▼ Developer (approve / reject / redirect via Matrix)

Why ZeroClaw

ZeroClaw is a Rust-based AI agent runtime that ships as a single binary under 9MB. It natively supports Matrix as a communication channel, has built-in SQLite memory with vector search, supports custom skills and tools with explicit allowlists, and can run as a daemon — all with under 5MB RAM. This means every agent in the pipeline is a lightweight ZeroClaw process with its own identity, skills, tool access, and memory, without writing custom Matrix integration code or building a framework from scratch.

Example Interaction

Developer types into Element on their phone:

"Add RFID tag reading to the door integration node. Use the serial protocol spec from the docs folder."
  1. Orchestrator (ZeroClaw on Matrix) receives the message, classifies it as a dev task, creates a task record, invokes the planner.
  2. Planner reads the referenced spec, analyzes the codebase, produces a structured plan. Orchestrator posts plan to Matrix for approval.
  3. Developer reviews and replies "👍" or provides adjustments.
  4. Coder creates a feature branch, writes the implementation, commits incrementally.
  5. Reviewer runs static analysis, checks conventions, reviews the diff. Posts summary.
  6. Tester runs the test suite in a clean environment, reports results.
  7. Orchestrator posts to Matrix: "Branch feature/rfid-door ready. 4/4 tests pass. Approve merge?"
  8. Developer replies "merge."
  9. Deployer merges, triggers CI, deploys to staging.
  10. Monitor watches the deployment, alerts if anything breaks.

Part 2

Agent Architecture

ZeroClaw as the Agent Runtime

Every agent in the system is a ZeroClaw instance. ZeroClaw provides the runtime layer that each agent needs: LLM provider calls (Anthropic, OpenAI, OpenRouter, Ollama), Matrix channel integration, persistent memory (SQLite + vector search), custom skill loading, tool execution with sandboxing, and daemon mode. You configure agents, not build them.

Key Insight

The orchestrator is itself a ZeroClaw instance connected to your Matrix project room. It receives your messages through ZeroClaw's native Matrix channel support and uses custom skills to classify intent, manage task state, and invoke other ZeroClaw agent instances. No separate matrix-sdk code or custom Rust binary needed for the base setup.

Agent Definition

Every agent is a ZeroClaw instance defined by:

  • Identity file — role, personality, constraints, output format (IDENTITY.md or AIEOS JSON)
  • Skills — capability modules dropped into the workspace directory
  • Tool allowlist — explicit list of what the agent can execute
  • Provider config — which LLM model to use and token limits
  • Channel config — which Matrix room(s) to listen on or report to

System Layout

Your server (NixOS on Hetzner) │ ├── zeroclaw-orchestrator (daemon) │ ├── channel: Matrix (your project room) │ ├── identity: orchestrator │ ├── skills: classify_intent, manage_tasks, spawn_agents │ ├── tools: sqlite, git_status, agent_invoke │ └── memory: task state, conversation history │ ├── zeroclaw-planner (invoked per task) │ ├── identity: planner │ ├── skills: analyze_codebase, create_plan │ ├── tools: file_read, git_log, tree │ └── provider: claude-sonnet │ ├── zeroclaw-coder (invoked per task) │ ├── identity: coder │ ├── skills: write_code, follow_conventions │ ├── tools: file_read, file_write, git, build, format │ └── provider: claude-sonnet │ ├── zeroclaw-reviewer (invoked per task) │ ├── identity: reviewer │ ├── skills: review_diff, check_conventions │ ├── tools: file_read, git_diff (read-only) │ └── provider: claude-sonnet │ ├── zeroclaw-tester (invoked per task) │ ├── identity: tester │ ├── skills: run_tests, summarize_failures │ ├── tools: test_runner, log_read │ └── provider: claude-haiku (cheaper, mostly tool execution) │ ├── zeroclaw-deployer (invoked per task) │ ├── identity: deployer │ ├── tools: git_merge, ci_trigger, deploy_script, rollback │ └── provider: claude-haiku │ └── zeroclaw-monitor (long-running daemon) ├── identity: monitor ├── skills: watch_logs, detect_anomalies ├── tools: log_read, metrics_query, alert └── channel: Matrix (alert room)

Agent Specifications

Orchestrator

A ZeroClaw daemon instance connected to your Matrix project room. Manages the task lifecycle.
How it works ZeroClaw's native Matrix channel receives your messages. The orchestrator's identity prompt and skills handle classification, task creation, agent invocation, and result routing. Task state is persisted in ZeroClaw's SQLite memory. Custom skills shell out to invoke other ZeroClaw instances and parse their output.
Skills classify_intent, manage_tasks, spawn_agents, check_status
State Machine
Received → Planning → Plan_Review
  → Approved → Coding → Code_Review
    → Reviewing → Testing
      → Ready → Merge_Review
        → Merged → Deploying → Deployed → Monitoring

Any state → Failed (with reason)
Any state → Cancelled (by developer)
Optional Extension

Hybrid Orchestrator

The base orchestrator uses ZeroClaw's built-in LLM provider to classify intent and route tasks — this is already "hybrid" by nature, since ZeroClaw agents use LLM calls as part of their normal operation. But for complex multi-project setups, you may want to replace the orchestrator's custom skills with a dedicated Rust binary that manages the state machine deterministically while still using ZeroClaw instances for each specialized agent.

When to graduate to a custom orchestrator:

  • You need fine-grained control over parallel task execution and resource limits
  • The skill-based approach becomes too fragile for complex state machine logic
  • You want typed, compile-time-checked task routing instead of LLM-driven classification
  • You're running 10+ concurrent tasks and need deterministic scheduling

Architecture with custom orchestrator:

Matrix message arrives │ ▼ Custom Rust orchestrator (matrix-sdk, typed state machine) │ ├── Classifier (LLM call via ZeroClaw — small, fast) │ ├── dev_task → standard pipeline │ ├── question → direct LLM response │ ├── ops_request → ops pipeline │ └── ambiguous → LLM decides routing │ ├── zeroclaw-planner (invoked as subprocess) ├── zeroclaw-coder ├── zeroclaw-reviewer └── zeroclaw-tester │ ▼ Deterministic state machine takes over

Cost control: Even in this model, ZeroClaw runs each agent. The custom binary only handles routing, state, and Matrix I/O — the LLM calls still happen inside ZeroClaw instances. This keeps the custom code focused and small.

Planner Agent

Decompose a high-level intent into a structured, ordered task list.
ZeroClaw tools file_read, git_log, tree (read-only)
Identity focus Senior technical lead. Given a feature request and project context, produce a step-by-step plan. Each step must be concrete and actionable. Identify files to create or modify. Flag risks and ambiguities.
Output
{
  "summary": "Add RFID serial reading to door integration",
  "steps": [
    { "id": 1, "action": "read_spec", "description": "Parse serial protocol" },
    { "id": 2, "action": "create_file", "path": "src/rfid_reader.cpp" },
    { "id": 3, "action": "modify_file", "path": "src/door_node.cpp" },
    { "id": 4, "action": "create_file", "path": "test/test_rfid.cpp" },
    { "id": 5, "action": "modify_file", "path": "CMakeLists.txt" }
  ],
  "risks": ["Serial baud rate not specified — need clarification"],
  "estimated_complexity": "medium"
}

Coder Agent

Execute implementation steps by writing and modifying code.
ZeroClaw tools file_read, file_write, git, build, format
Identity focus Expert developer. Follow conventions exactly. Clean, documented code. Commit after each logical change. Never modify files outside task scope.
Constraint Tool allowlist excludes deploy and remote access. Operates exclusively in a feature branch.

Reviewer Agent

Review code changes for correctness, style, and safety.
ZeroClaw tools file_read, git_diff (read-only)
Identity focus Thorough code reviewer. Check for bugs, security issues, error handling gaps, race conditions. Categorize as blocking / warning / suggestion.
Constraint Tool allowlist is read-only. Cannot modify code. Blocking issues loop back to coder.
Output
{
  "verdict": "pass_with_warnings",
  "issues": [
    { "severity": "warning", "file": "src/rfid_reader.cpp",
      "line": 42, "message": "Missing timeout on serial read" }
  ]
}

Test Agent

Execute test suites and report results.
ZeroClaw tools test_runner, log_read
Workflow Run tests in clean Nix dev shell → parse results → LLM summarizes failures → report back.
Provider Uses a cheaper/faster model (e.g. Haiku) since this agent is mostly executing tools and parsing output.

Deploy Agent

Execute deployment workflows.
ZeroClaw tools git_merge, ci_trigger, deploy_script, rollback
Constraint Always requires human approval for production. Staging can be configured as auto.

Monitor Agent

Watch runtime health after deployment.
Implementation Runs as a long-lived ZeroClaw daemon (not invoked per-task). Watches logs, metrics, health endpoints. Uses LLM to interpret anomalies and decide whether to alert. Posts to a dedicated Matrix alert room.

Context Pipeline

The orchestrator assembles context for each agent. Each agent sees only what it needs — previous agent outputs flow through the orchestrator, not directly between agents.

Planner output (task list) → orchestrator feeds into Coder as instructions → Coder output (branch, diff, commits) → orchestrator feeds into Reviewer as review target → Reviewer verdict → if pass → Test Agent → if fail → loop back to Coder with issues → Test result → if pass → ready for merge approval → if fail → loop back to Coder

Security Model

ZeroClaw's built-in security features map directly to the agent architecture:

  • Tool allowlists — each agent's config.toml specifies exactly which tools it can use. The coder cannot deploy; the reviewer cannot write files.
  • Workspace isolation — each agent operates in its own scoped workspace directory
  • Encrypted secrets — API keys and tokens are encrypted at rest by ZeroClaw
  • Sandboxed execution — ZeroClaw supports native and Docker sandboxing with configurable network, memory, and CPU limits
  • All actions logged with agent identity and timestamp via ZeroClaw's observability layer

Part 3

Infrastructure

Communication — Matrix/Synapse

Matrix serves as the universal message bus, accessed natively through ZeroClaw's built-in Matrix channel support. Self-hosted Synapse gives full data ownership, E2EE for sensitive projects, and a rich client ecosystem.

  • Project room — the orchestrator listens here for developer messages
  • Admin room — agent management commands
  • Alert room — monitor agent posts deployment health
  • Archive — full audit trail of every decision and action
ZeroClaw + Matrix

ZeroClaw's Matrix channel integration handles authentication, message parsing, and response posting out of the box. The orchestrator agent simply receives messages as input and posts responses as output — no custom Matrix client code needed. Additional agents communicate with the orchestrator through ZeroClaw's tool/skill invocation, not through Matrix directly.

Execution — NixOS

All agent code execution happens inside Nix dev shells: reproducible, isolated, declarative, and fast after first build.

{
  devShells.default = pkgs.mkShell {
    buildInputs = with pkgs; [
      gcc cmake colcon ros2-humble clang-tools
    ];
    shellHook = ''
      export ROS_DOMAIN_ID=42
      source /opt/ros/humble/setup.bash
    '';
  };
}

Agent tools execute within nix develop --command <tool> for each step, ensuring consistent environments.

Hosting & Networking

  • Primary — dedicated server (e.g. Hetzner) running NixOS
  • Services — Synapse, ZeroClaw daemons (orchestrator + monitor), agent workspaces
  • Networking — Tailscale for secure access from any device
  • Monitoring — ZeroClaw's built-in observability + Prometheus/Grafana for infrastructure
  • Mobile client — Element on Android/iOS

Resource Footprint

ZeroClaw's minimal resource usage makes this architecture feasible on modest hardware. The orchestrator and monitor run as persistent daemons (under 5MB RAM each). All other agents are invoked on demand and exit after completing their task. A single Hetzner VPS can comfortably run the entire system.

Part 4

Implementation Plan

Phase 0 Immediate Productivity Week 1

Goal: Productive remote dev from phone today, zero custom infrastructure.

Install Termux + Tailscale on Android. SSH into desktop, start tmux session. Use an AI coding tool interactively. Optional: Mosh for better mobile resilience.

Deliverable: Working SSH-based dev workflow from phone.

Phase 1 ZeroClaw on Matrix — Single Agent Weeks 2–4

Goal: Send a coding request via Matrix, get a branch with changes back.

Install ZeroClaw on the server. Configure a single agent with Matrix channel, an identity prompt for coding tasks, and tools for file I/O and git. The agent receives messages from Element, writes code in a feature branch, and reports back. Task history is stored in ZeroClaw's built-in SQLite memory.

No multi-agent separation yet — one ZeroClaw instance handles everything in a single pass.

Deliverable: Type a request in Element, get a branch with a diff link.

Phase 2 Orchestrator + Planner/Coder Split Weeks 5–8

Goal: Separate planning from execution, add human oversight.

Promote the single agent to an orchestrator role. Add custom skills for intent classification and task management. Create separate ZeroClaw workspaces for the planner and coder agents. The orchestrator invokes them via skills and posts plans to Matrix for approval before coding begins.

Deliverable: Two-phase workflow — plan review, then code generation.

Phase 3 Automated Review + Testing Weeks 9–12

Goal: Automated quality gates before results reach the developer.

Add reviewer and tester ZeroClaw instances. Reviewer runs linting + LLM diff review. Tester executes the suite in a Nix dev shell. Orchestrator manages failure loops back to coder. Max retry limit (default 3) to prevent infinite loops.

Deliverable: Developer only sees changes that pass lint, review, and tests.

Phase 4 Deploy + Monitor Weeks 13–16

Goal: End-to-end from intent to production.

Add deployer and monitor ZeroClaw instances. Monitor runs as a persistent daemon watching logs and metrics, posting to a dedicated Matrix alert room. Rollback capability via !deploy rollback. Staging = auto, production = always ask.

Deliverable: Full loop — type intent, approve plan, approve merge, monitor deployment.

Phase 5 Multi-Task + Parallelism Weeks 17–20

Goal: Handle multiple concurrent tasks across projects.

Task queue with priority and dependencies. Per-project agent configurations. Concurrent ZeroClaw invocations on different branches. Resource management for LLM API rate limits and git lock contention. Evaluate whether to graduate to a custom Rust orchestrator (see Optional Extension above).

Deliverable: Work on multiple features simultaneously from your phone.

Phase 6 Refinement + Prompt Engineering Ongoing

Goal: Continuously improve agent quality.

Version identity files and skills in git. A/B test prompt variations on non-critical tasks. Build project-specific context libraries. Analyze failure modes via ZeroClaw's observability logs. Add specialized sub-agents for specific domains (database migrations, API design, documentation).

Agent Management

Admin Commands (Matrix)

The orchestrator responds to admin commands in the Matrix admin room:

!agents list Show all agents and status !agents reload <agent> Hot-reload agent identity/skills !agents disable <agent> Temporarily disable !agents enable <agent> Re-enable !agents logs <agent> --last <n> Recent actions !task list Show active tasks !task status <id> Detailed task status !task cancel <id> Cancel running task !task retry <id> Retry from last good state !task history <id> Full task timeline !config gate <agent> <action> auto|ask Change gate settings

Identity & Skill Versioning

Agent identities and skills are stored in git alongside the project:

agents/
  orchestrator/
    IDENTITY.md
    skills/
      classify_intent/
      manage_tasks/
    CHANGELOG.md

  coder/
    IDENTITY.md       ← v2 (current)
    IDENTITY.v1.md    ← previous version
    skills/
      write_code/
    context/
      CONVENTIONS.md

  reviewer/
    IDENTITY.md
    skills/
      review_diff/

Rollback is swapping an identity file. The orchestrator's !agents reload command picks up changes without restarting the daemon.
